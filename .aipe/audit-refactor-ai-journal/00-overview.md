# Chapter 00 — Overview

## The codebase in one paragraph

buffr is a solo-developer, Android-only daily-journaling app — free-form prose plus structured "drops" (`[]` todos, `** food N kcal` nutrition lines, `#tag` thread mentions), a habits/cadence grid, and a vlog editor. The stack is React Native 0.83 + Expo SDK 55 + TypeScript strict, with `expo-sqlite` as the local canonical store (file `buffr.db`, WAL mode), Supabase Postgres as a cloud mirror under the `buffr` schema (since migration 0010 on 2026-05-19), and a 5-chain AI service that runs against Anthropic by default and OpenAI as the alternate. Single user, single device, pre-Phase-B auth (a hardcoded `PHASE_A_USER_ID` UUID stamps every cloud row). Release builds sign with the debug keystore and ship via `adb install -r`; no Play Store, no second device.

## The staff-engineer's overall take

**Take:** buffr is a small codebase that knows what it is. The four load-bearing principles in `docs/spec.md` §10 — "DB is single source of truth," "prose is canonical for drops," "two-pass matching," "cloud is a sync mirror, never the canonical source" — actually thread through every module that touches data. Most codebases at this size have one or two principles in the README and ignore them in practice; this one enforces them, and it shows in the sync engine being twelve files, the scanners running only at commit instead of per-keystroke, and the cloud schema being multi-tenant-ready from day one even though there's only one user. The interesting refactor opportunities aren't in the data layer — they're in the AI layer and in two specific places where the same concept has spread across multiple files without anyone noticing.

**Take:** the pattern of weaknesses is "things that worked the first time someone needed them, and were never revisited because the codebase grew sideways instead of deeper." Five AI chains all carry the same `provider === 'openai' ? ... : ...` branching boilerplate because the first chain (probably `summarize`) was authored before the second; by the time there were three chains the cost of refactoring to a Strategy was unclear, and now there are five. Clip handling is in five files because each new clip-related need landed where it was most convenient (`fileManager.ts` for the new path, `clipMigration.ts` for the rename story, `database.ts:repairBareClipUris` for the bare-filename bug, two mapper files for sync) — no one drew the box. The 1456-line `database.ts` is what every "single mouth to SQLite" codebase eventually has: the discipline of one writer becomes the cost of one massive file.

**Take:** what I'd be nervous about maintaining is **observability under failure**. The sync orchestrator's `if (applied > 0 || fetched > 0)` log guard swallowed every PGRST106 error for an hour on 2026-05-19 — diagnosis took a `curl` round-trip from the laptop because nothing in logcat said anything was wrong. The cleanup audit caught this and there's a fix-now spec for it, but the underlying shape (sync failures land silently in `sync_meta.last_error` with no console signal, no UI surface, no alarm) is the pattern. Build one more silent failure mode in this codebase and a real user-data bug will live for weeks. The fix isn't a refactor — it's a feature-level decision about how the app reports its own internal state to the developer.

**The one thing I'd change if I could only change one thing:** consolidate the AI-provider switch into a Strategy. Five chains × two providers × identical branching logic = the single largest piece of meaningful duplication in the codebase, and it's the one that compounds — the day a third provider lands (Gemini, a local Llama, doesn't matter), the cost of adding it is editing five files and re-testing every chain. The refactor is small (≤150 lines net), the principle violation is real (Open/Closed, DRY-with-care, Single Responsibility per chain file), and the runtime behaviour stays identical. Everything else in this book is either smaller in scope or larger in risk.

## Reading order and chapter summaries

**Chapter 01 — Composition.** One DEEP section (Replace Conditional with Dispatch Table for the AI-provider switch), two BRIEF, one MENTION. Verdict on the DEEP: *worth doing.* The chapter as a whole says buffr's composition health is good — most functions are sized right, names are honest, the rename refactor from loopd→buffr earlier this month went cleanly because the underlying naming was clean to begin with.

**Chapter 02 — Structural.** Two DEEP sections: Separate Pure from Effectful (the AI chains mix prompt-shaping with API calls), and Extract Module (the five-file clip-handling sprawl). One BRIEF on Introduce Boundary for the AI providers. Verdict: *the AI separation is worth doing eventually; the clip extraction is worth doing only if a new clip-feature lands.* Chapter pattern: the codebase has good intra-module cohesion and respectable boundaries between domains; the structural debt is at one specific cross-cutting concern (clips) that grew before anyone defined a box for it.

**Chapter 03 — Patterns.** One DEEP section (Strategy for AI providers — same locus as Chapter 01's DEEP, viewed through the pattern lens), one BRIEF (Facade — noting that the sync orchestrator is already one and that's a good thing). Most named patterns don't apply here. Verdict: *Strategy is worth doing.* Chapter pattern: this codebase doesn't need most patterns, and that's correct — applied patterns without underlying problems are how small codebases get over-engineered. The one pattern it actually needs has been deferred.

**Chapter 04 — DSA.** One BRIEF (Collapse Traversals on the commit-time scanner triplet), one BRIEF (Memoize cellState in the daily-schedule grid). No DEEP sections — and the chapter intro says so. Verdict: *not worth doing today; the codebase doesn't have a DSA problem at single-user scale.* Chapter pattern: the architectural principle that scanners run only at commit and that sync paginates kept the DSA surface tiny by design.

**Chapter 05 — Principles.** Three DEEP principles: Open/Closed (provider extensibility — same locus, third lens), Locality of Behaviour (clip handling scattered across five files), and Single Responsibility (`database.ts` at 1456 lines doing schema + CRUD + repair + sync hooks). Brief treatment of Separation of Concerns, Tell-Don't-Ask, Principle of Least Surprise. The chapter's closing paragraph is probably the most useful one in the book: it names which principles buffr honours by default (Tell-Don't-Ask, Locality of Behaviour within a domain, Separation of Concerns across the four data principles) and which it strains against (Open/Closed for provider extensibility, Single Responsibility for the database mouth, Locality of Behaviour across the clip concept).

**Recommended reading order:** start with Chapter 05 — the principles chapter does the most to surface what this codebase is actually doing well versus where it's straining, and the other chapters' DEEP sections all map back to one of three principles. After 05, read 03 (Strategy is the cleanest single move you could make), then 02 (the structural takes are the longest), then 01 (composition is the smallest scope), then 04 (which is mostly NOT FOUND items confirming the codebase doesn't have DSA debt). 00 is the meta-take you're reading now — return to it once for the one-thing-I'd-change reminder.
