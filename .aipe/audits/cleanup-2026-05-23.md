## Lens 1 — Structural (code health)

### Silent sync failures: orchestrator log-guard swallows error results

**Lens:** structural
**Severity:** high
**Effort:** S
**Decision:** fix-now
**Refactor-shape:** in `src/services/sync/orchestrator.ts` add an `else if (r.error)` branch after each `if (r.applied > 0 || r.fetched > 0)` log inside `pullAll()` (line 72) and `pushAll()` (line 49). The branch emits `console.warn('[buffr sync] pull <table> failed:', r.error)` (and the push equivalent). No control flow changes; same return shape; same `recordSyncError` write into `sync_meta`. Only the console gains visibility for the failure path that already exists.

**Background:** the 2026-05-19 schema namespace move (migration 0010) broke every sync call for hours with zero logcat signal because PGRST106 errors come back as `{fetched: 0, applied: 0, error: '…'}` and the guard suppresses them. Diagnosis required a `curl` round-trip from the laptop. The orchestrator stores the error in `sync_meta.last_error` but never surfaces it to the dev console. This is the lowest-cost behaviour-preserving change with the highest catch-rate for the next silent-failure mode.

### Dead schema columns + `sync_deletions` table linger from deleted Notion sync

**Lens:** structural
**Severity:** medium
**Effort:** M
**Decision:** fix-later
**Refactor-shape:** author Supabase migration `0011_drop_dead_columns.sql` with `ALTER TABLE … DROP COLUMN IF EXISTS` for `notion_page_id` + `notion_last_synced` on `entries`/`nutrition`/`habits`/`threads`, plus `todo_meta.stage` + `todo_meta.position`, plus `DROP TABLE IF EXISTS sync_deletions`. In `src/services/database.ts`'s `migrate()`, add a SecureStore-gated `ALTER TABLE … DROP COLUMN` pass for local SQLite (3.35+ supports it via expo-sqlite), drop the dead lines from `CREATE TABLE` blocks, and remove the now-stale `idx_*_notion` indexes. Grep `src/services/sync/tables/*` for any remaining mapper references and strip them. Behaviour stays identical because nothing reads these columns or the table.

**Background:** Notion sync was deleted some time ago but its outbox (`sync_deletions`) and per-row pointer columns (`notion_page_id`, `notion_last_synced`) survived on entry tables, plus index entries. `todo_meta.stage` was replaced by `done` on 2026-05-05; `todo_meta.position` was replaced by `pinned` the same day. All round-trip through cloud sync but no read path consults them. Not blocking anything — purely cosmetic + onboarding tax.

### `clip_uri` / `clips_json` legacy single-clip column coexists with multi-clip array

**Lens:** structural
**Severity:** medium
**Effort:** M
**Decision:** fix-later
**Refactor-shape:** SecureStore-gated one-time backfill in `src/services/database.ts`: walk `entries WHERE clip_uri IS NOT NULL AND (clips_json IS NULL OR clips_json = '[]')`, set `clips_json = JSON_ARRAY(JSON_OBJECT('uri', clip_uri, 'durationMs', clip_duration_ms))`. Mark backfill flag. Follow-up Supabase migration drops `clip_uri` + `clip_duration_ms` from local and cloud schemas. Read sites in `src/services/database.ts` + `src/components/journal/*` switch to `clips_json` only, removing the existing `if (!clips_json) return [{uri: clip_uri, ...}]` fallbacks.

**Background:** the multi-clip feature was added later. Both columns coexist because the old column is the back-compat pointer for pre-migration rows. Every read site that wants "all clips" carries an `if (clips_json empty) fall back to clip_uri` branch. New contributors miss the fallback and ship a bug visible only on old rows. Fix-later because the backfill is one-time and the current code is correct.

## Lens 3 — DSA hot-paths

### No DSA-shaped debt this pass

Scanners (`scanTodos`, `scanNutrition`, `scanThreads`) only run at commit (focus blur / screen leave) per the architectural principle; not per-keystroke. Push batches at 50; pull pages at 200; ffmpeg concurrency caps at 2. `clipMigration` is O(N) on clip count per cold start (~183 entries today) — bounded and not on a hot path. `repairBareClipUris` runs in background after DB open and walks `entries WHERE clips_json IS NOT NULL` once. Nothing in the codebase is doing nested scans of grown data or computing a derived value per render where a memo would help. The codebase doesn't have a DSA hot-path problem today.

## Lens 4 — Test debt

### No automated test suite covering load-bearing logic

**Lens:** test
**Severity:** high (in absolute terms)
**Effort:** L
**Decision:** accept
**Refactor-shape:** N/A — accepted decision per `.aipe/project/rules.md` ("No automated test suite at present. Manual end-to-end on the connected Android device after each meaningful change").

**Background:** scanners (`scanTodos`, `scanNutrition`, `scanThreads`), reconciler (`reconcileMeta`), conflict resolver (`chooseWinner`), heuristic classifier, sync mappers — all uncovered. A regression in any of these would only surface when a user notices broken data in the UI. `npx tsc --noEmit` catches type-shape regressions but not behavioural ones. The explicit project decision is that single-developer pace + RN-UI test cost + weekly-changing feature surface outweighs the regression-catch value at this stage. **Worth re-visiting when the codebase opens to other users (Phase B) or when manual testing misses a regression that costs real data.** Until then: accept.

## Cannot-clean (require behaviour change → exit cleanup, file as feature work)

These came up in the scan but every one of them requires changing what the app does, not just how it's organized. Filing here so they're not lost.

### Silent sync failures need a UI surface in `app/settings/cloud-sync.tsx`

The console-warn fix above (fix-now Lens 1) catches errors during development. For a user-facing app, `sync_meta.last_error` should be visible in the cloud-sync settings screen — "Last sync error: PGRST106 (2 min ago)". That's adding a new UI capability, not preserving behaviour. **File as: `feature/cloud-sync-error-surface`.**

### Stale clip files from loopd → buffr sandbox boundary

The 183 broken clip references could be (a) auto-detected and the entries' `clips_json` field cleared, (b) presented to the user as "we couldn't find these clips — remove them?", or (c) left as-is. All three are behaviour changes (data deletion or new UI). **File as: `feature/orphan-clip-reconcile`.**

### Phase A → Phase B auth migration

Replacing hardcoded `PHASE_A_USER_ID` with `auth.uid()` requires real Supabase Auth + RLS activation + onboarding + payment + a one-time SQL update to re-stamp every existing cloud row with the actual signed-in user's UUID. Multi-week work, sketched in `docs/buffr-cloud-sync-spec.md` §14. **File as: `feature/phase-b-auth`.**

### `interpret` chain output not persisted

Currently modal-only — closing the modal loses the markdown, and re-running costs another LLM call. Persisting it (cache by date? store in `ai_summaries` alongside the structured summary?) is a feature decision. **File as: `feature/interpret-persistence`.**

### Vacuum job + clip file backup

Both named in `docs/backlog.md`. Hard-deleting `deleted_at < NOW() - 30d` rows is a behaviour change (data removal). Uploading clip bytes to Supabase Storage is a new capability + cost surface. **File as: `feature/sync-vacuum` and `feature/clip-cloud-backup`.**
