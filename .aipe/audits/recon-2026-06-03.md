# Recon — AI-engineering readiness for buffr (2026-06-03)

## 1. AUDIT — the AI/ML surface as-built

**Shape:** LLM-application engineering (per the three-shapes framing in `study-ai-engineering`). Not classical ML. Not RAG (deliberately — `docs/spec.md` principle #11 says no RAG until provably needed).

**The chains (`src/services/ai/`):**

| Chain | Purpose | Frequency | Evidence |
|---|---|---|---|
| summarize | one-paragraph day summary | 1×/prose-commit | the prompt + provider call |
| caption | image-grounded caption | 1×/upload (cached) | OpenAI multimodal call |
| expand | terse → prose | on-demand (rare) | `src/services/ai/expand.ts` |
| classify | classify a candidate line into `todo \| idea \| knowledge \| study \| reflect` | fan-out per prose-commit (heuristic ~70%) | classifier with heuristic short-circuit |
| interpret | thread-level interpretation | per impacted thread per commit | `src/services/ai/interpret.ts` |

**The orchestration (`src/services/prose/compose.ts`):** deterministic outer function that fans out chain calls through `cachedChain(...)` — content-hashed cache in `ai_summaries`. Compose itself is pure with respect to LLM side effects: same input → same output (modulo cache state).

**Provider abstraction:** Anthropic primary, OpenAI fallback + always-on-for-multimodal. Single facade in `src/services/ai/`. Manual failover today.

**What's hand-built (high authorship density):**

- `compose.ts` — the orchestrator pattern (`study-system-design/03`).
- `cachedChain` — the cache short-circuit (`study-system-design/03`, `study-performance-engineering/01`).
- The heuristic-before-LLM classifier (`study-system-design/05`) — ~70% short-circuit rate.
- `reconcileMeta.ts` — the multi-table prose-projection reconcile in one local SQLite txn.
- Provider abstraction over both Anthropic + OpenAI.

**What's NOT in the repo (the three usually-missing things, checked explicitly):**

- **EVALS — absent.** No `tests/evals/`, no `LLM-as-judge` evaluator anywhere, no golden eval set, no regression set per chain, no agreement metric, no eval cron, no eval in CI. The chains' correctness is "the user re-saves if it drifts." (`study-testing/06-testing-ai-features.md` already names this gap; recon confirms it.)
- **FAILURE HANDLING — partial.** Chains throw on validation failure (`cachedChain` validates before cache write — good). Compose catches and skips derivation gracefully — good. But there is **no retry-on-invalid-output**, no repair-step on malformed JSON, no provider auto-failover on transient Anthropic errors (`study-distributed-systems/02` + `study-networking/08` both name this), no abstention mechanism.
- **TRADEOFF EVIDENCE — partial.** `docs/spec.md` documents *some* tradeoffs (no RAG, prose-as-source-of-truth). What's missing per-chain: "we picked structured-output via tool calls because… we rejected JSON-mode because…" — that level of decision evidence is not in code or commit messages.

**The repo's true level: L1.**

Median of claimed competencies is L1 (most chains, foundations, prompts). It's dragged down by L0 on evals — and the spec's scoring rule (no eval = cap at L1) means even the L2 competencies cannot read as L2 in a room without the eval evidence to back them up. Heuristic-before-LLM at L2 is genuine; caching at L2 is genuine.

**Claim-vs-reality.** A resume that says "shipped LLM application with prompt chaining, provider abstraction, content-hashed cache, on-device sync" reads as L2-leaning. Under interview push-back ("how do you know the classifier is good?"), the answer collapses to "the user can re-save if it drifts" — a clean L1 admission and the thread an interviewer pulls.

## 3. PATH — the track this repo advances

**The track:** LLM-application engineering. Single track, well-shaped, but capped.

**The strongest signal latent in the repo (the centerpiece if pushed to L3):** the **cache short-circuit as cost ceiling** pattern. It is genuinely novel-ish, defensible, and has a clear story shape. Pushed to L3 — eval-backed cache-hit-rate as a regression metric, with a documented prompt_version invalidation discipline — it becomes an interview centerpiece. The two pattern files in `study-performance-engineering/` and `study-system-design/03` already do the architectural framing; what's missing is one measurement and one war story.

**The load-bearing gap:** **no evals.** Capping every track at L1 and the centerpiece at L2. Closing this one rung (build an LLM-as-judge for one chain; pick classify because the output is discrete and labelable) lifts the ceiling on every chain-related competency at once. This is the single highest-leverage move.

**Claim-vs-reality summary:**

| What it reads as on a resume | Where it falls apart under questioning |
|---|---|
| "Shipped LLM application, 5 chains, content-hashed cache" | "How do you know the classify chain is correct?" → "the user re-saves if it drifts" → L1 admission |
| "Heuristic-before-LLM saves ~70% of classifier calls" | "What's your false-positive rate on the heuristic?" → "haven't measured" → L1 admission |
| "Designed for cost ceiling under unique-input growth" | "What's your cache hit rate?" → "not measured" → L1 admission |
| "Provider abstraction over Anthropic + OpenAI" | "What's your failover behavior on a 503?" → "manual" → L1 admission |

Every push-back lands on the same gap: **no measurement, no defended number.**
