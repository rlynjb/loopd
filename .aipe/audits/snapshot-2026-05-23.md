# buffr — audit snapshot, 2026-05-23

## 1. Identity

buffr is a solo-developer, native-Android daily-journaling app that combines free-form prose entries with structured "drops" (`[]` todos, `** food N kcal` nutrition lines, `#tag` thread mentions), a habits/cadence grid, and a vlog editor that trims and exports clips with text + filter overlays. Beneath the journal sits an AI layer that classifies each todo into a "thinking mode" (todo / idea / knowledge / study / reflect), expands selected todos into structured markdown, generates four tonal caption variants per day, and produces a long-form "interpret" reflection on demand.

The primary user is the developer himself — one person, one phone, one cloud account. There is no signup, no auth screen, no second device today; a hardcoded `PHASE_A_USER_ID` UUID stands in everywhere the system would otherwise call `auth.uid()`.

Stage: **in active use, single-user**. The codebase is past prototype — every day's writing happens in it — but it is pre-multi-user. Release builds are signed with the debug keystore and sideloaded via `adb install -r`; no Play Store listing, no end-user distribution channel.

## 3. Architecture

**Top-level directories** (excluding `android/`, `node_modules/`, build artifacts):
- `app/` — file-routed expo-router screens. `_layout.tsx` is the boot path (orchestrates SQLite open, AI auto-summary, the seven one-time backfills, and the cloud-sync `bootstrap`/`pullAll`/`pushAll` cascade).
- `src/services/` — all domain logic, split by area: `ai/`, `todos/`, `threads/`, `nutrition/`, `habits/`, `sync/`, plus root files (`database.ts`, `fileManager.ts`, `ffmpeg.ts`, `ffmpegCommand.ts`, `clipMigration.ts`, `exportPipeline.ts`, `textBitmap.ts`, `textRenderer.tsx`).
- `src/components/` — UI subtrees: `home/`, `journal/`, `editor/`, `todos/`, `threads/`, `nav/`, `timeline/`, `ui/`.
- `src/hooks/`, `src/types/`, `src/constants/`, `src/utils/` — typed glue + theme tokens.
- `supabase/migrations/` — append-only Postgres DDL (10 files as of 2026-05-19).
- `scripts/` — the migration runner.
- `docs/` — `spec.md` is canonical; the rest are design/plan docs.
- `assets/` — fonts, icon variants, splash.
- `.aipe/` — agent state: `project/` (context, rules, stack), `study-ai-journal/` (80-file diagrams-first study guide), `specs/interview/` (interview-prep chapters), `audits/` (this file).

**State model** — three concentric rings:

```
   Local SQLite (canonical)          ← every read, every write
       │
       │  schedulePush() debounced 5s
       ▼
   Supabase Postgres (buffr schema)  ← durability + future multi-device
       │
       │  pull on app open + on demand
       ▼
   Local SQLite                       (closes the loop via firstPull on fresh device)

   Files: Documents/buffr/{media,exports,temp}/   (device-local, not synced)
   Secrets: Android Keystore via expo-secure-store
```

`database.ts` is the only file that opens `buffr.db`. Every write calls `schedulePush()` so the cloud catches up. Reads never round-trip to the cloud.

**Sync flow** — push and pull are independent flows over the same 10-table `REGISTRY` defined in `src/services/sync/orchestrator.ts`:
- `pushTable()` (push.ts): `SELECT * WHERE updated_at > synced_at`, batches of 50, upserts via `.upsert(rows, { onConflict: 'user_id,id' })`, stamps `synced_at` on success. Failed batches stay dirty and retry on the next push.
- `pullTable()` (pull.ts): server-time RPC → `WHERE updated_at > last_pull_at`, pages of 200 in `updated_at ASC`, per-row `chooseWinner(local, cloud)` (last-write-wins by `updated_at`), upserts winning row + stamps `synced_at = serverTime`.
- `bootstrap.ts` decides between initial-push, first-pull, no-op, or both-populated-fallback on first cold start; gated by SecureStore `cloud_initial_push_done`.
- `firstPull.ts` (30 LOC) resets every `sync_meta.last_pull_at` to NULL, then delegates to `pullAll()` — special case as parameter, not as fork.

**External integrations:**
- **Supabase Postgres** — cloud mirror; `buffr` schema (post-0010), exposed via PostgREST. Auth-tables stay in `auth` schema (Supabase-managed).
- **Anthropic + OpenAI** — LLM APIs. Provider switch in `services/ai/config.ts`; keys in SecureStore. Five chains × 2 providers = 10 explicit branches.
- **Android MediaStore / DCIM** — `saveToDCIMBuffr()` in `fileManager.ts` writes exported vlogs to the gallery.
- **EAS Update endpoint** — wired in `app.json` (`updates.url`, project ID `a5d9b9e0-…`), never used.

**Boundaries:**
- UI (`app/*`, `components/*`) reads via hooks (`useEntries`, `useDayTitle`, `useHabits`, etc.) and writes via service-layer functions. Hooks never speak SQL.
- Services (`src/services/*`) own SQL, AI calls, ffmpeg, sync. The five "Phase A architectural non-negotiables" in `docs/spec.md` §10 are enforced here.
- Sync (`src/services/sync/*`) is a self-contained layer; nothing outside it touches Supabase directly.
- AI providers are abstracted behind a per-chain function (`summarize`, `caption`, etc.) that internally switches on provider — no callsite chooses Claude vs GPT.

## 4. Features

Grouped by area, most stable first.

**Daily journal (stable).** `app/journal/[date].tsx` — free-form prose per date, autosaves on every keystroke to `entries.text`. Scanners (`scanTodos`, `scanNutrition`, `scanThreads`) run only at commit (focus blur, screen leave) per the "DB-first autosave / scanners at commit" principle. Two-pass matching (exact → line-index fallback) preserves row identity across edits.

**Today dashboard (stable).** `app/index.tsx` — `HomeHeader` (date, AI summary card), `SmartTodoList` (pinned-first then `createdAt DESC`), `DailyScheduleGrid` (weekly grid of habits with off-day toggle, locked to current week as of 2026-05-08), `PastVlogCard`. The threads + per-row x were dropped 2026-05-08.

**Todos (stable).** `app/todos.tsx` + `app/todos/[id].tsx`. Pinned-first sort. Each todo has a `todo_meta` row (1:1 invariant, enforced by `reconcileMeta`). Six thinking modes in CHECK (`todo`, `idea`, `knowledge`, `study`, `reflect`, `reduce`) after the 0006–0008 reductions/additions. `user_overridden_type` lock pins user-chosen classifications across re-classify passes.

**Threads — `#tag` project attribution (stable).** `app/threads/[id].tsx` + `app/more/threads.tsx`. Slug-as-local-canonical. Junction `thread_mentions` carries one row per `#tag` occurrence; one documented deviation (`touch.toggleThreadTouchToday`) writes both `entry_id` and `todo_id` NULL to mark "done today" from the dashboard.

**Nutrition (stable).** `app/more/nutrition.tsx`. One row per `** food N kcal` line in prose. Same two-pass scanner pattern.

**Habits + cadence (stable).** `app/more/habits.tsx`. `cadence_type` (daily / weekly / monthly), `cadence_days`, `cadence_count`, `time_of_day` (morning / midday / evening / anytime). Pure cadence math in `services/habits/cadence.ts` (`isDueOn`, `needsMoreThisWeek`, `isoWeekDates`).

**AI summaries + 4-variant captions (stable).** `services/ai/summarize.ts` + `caption.ts`. Cached per date in `ai_summaries.summary_json`. Captions emit four tonal variants (`clean` / `smoother` / `reflective` / `punchy`) plus a `variantsTheme`. Auto-runs on app open for yesterday if missing.

**Interpret modal (stable, newer — 2026-05-10).** `services/ai/interpret.ts`. Long-form markdown reflection on demand from the journal screen. Only AI chain that emits markdown rather than JSON; only chain whose output is not persisted (modal-only, gone on close).

**Vlog editor + export (stable).** `app/editor/[date].tsx` — trim, multi-clip, text overlays, filter overlays. `services/exportPipeline.ts` + `ffmpeg.ts` transcode to 1080p H.264. Concurrent transcodes capped at 2 (FIFO queue). Disk-full + cancellation errors are typed (`TranscodeCancelledError`, `DiskFullError`).

**Cloud sync (stable).** Local SQLite canonical, Supabase Postgres mirror. Push debounced 5s; pull on cold start + on demand. Bootstrap auto-detects initial-push vs first-pull vs no-op. As of 2026-05-19 (migration 0010) all 10 mirrored tables live in the `buffr` schema; client default schema is `buffr` so call shapes are unchanged. Verified end-to-end on device after the schema move.

**Per-todo AI expansion (stable).** `services/todos/expand.ts` + `expandPrompts.ts` + `expandSerialize.ts`. Sonnet/4o generates a typed structured-output payload per thinking-mode (4 schemas — `idea`, `knowledge`, `study`, `reflect`; `todo` is the non-expandable default). Result persisted as `todo_meta.expanded_md`.

**Heuristic-first classifier (stable).** `services/todos/heuristicClassify.ts` runs regex/keyword rules per line before calling the LLM. Confident heuristic matches skip the network call; ambiguous lines fall through to Haiku/4o-mini.

**Settings screens (stable).** `app/settings/{ai,cloud-sync,index,updates}.tsx`. AI provider toggle, API-key entry, cloud-sync diagnostics, manual update check (Updates page is mechanically working but always reports "latest" — see Section 6).

**App-icon + name + URL scheme (recently changed, stable).** Lowercase `b` in Nunito Regular at ~90% of the prior cap-height, beige `#E8D5B0` on black. Android applicationId `com.anonymous.buffr`, scheme `buffr://`.

## 5. Decisions

**Local SQLite is canonical; cloud is a mirror.** Every read hits `buffr.db`. Writes commit locally first, then `schedulePush()` (debounced 5s) sends the dirty rows to Supabase. Reasoning: keystroke→UI latency must be 1ms, not a network round-trip; the app must remain useful with no connectivity; conflict resolution is bounded because there's one writer per row at single-user scale. Cost accepted: every synced table carries `synced_at` + `deleted_at` and the codebase carries ~12 files in `src/services/sync/` (orchestrator, push, pull, conflict, bootstrap, firstPull, schedulePush, syncMeta, devActions, types, tables/*). Explicitly rejected: cloud-canonical (would gate every read on network), and a CRDT layer (overkill for single-writer per row).

**Prose is canonical for drops.** `[]`, `** food N kcal`, and `#tag` mentions in `entries.text` are the source. Derived tables (`todos_json` field, `nutrition`, `thread_mentions`) are rebuilt at commit time by scanning prose. Reasoning: a freeform editor with structured-record output is impossible if the user has to maintain both; the prose stays the user's mental model, and the typed records are the machine's. Cost: every prose-derived feature ships a SecureStore-gated one-time backfill (run once per install) and a two-pass matching algorithm. Rejected: parallel structured editors per drop type.

**Composite `(user_id, id)` PKs on every synced table.** Schema-level isolation that holds today (Phase A, hardcoded user_id) and after RLS ships (Phase B, real auth). Reasoning: defense in depth — even a misconfigured RLS policy or a leaked anon key can't return another user's row because the row's full key includes a `user_id` the caller doesn't know. The PK shape was decided once, on day one, so Phase B activation is `auth.uid()` replacing the hardcoded UUID — not a schema rewrite. Rejected: single-column PKs with RLS-only enforcement.

**RLS scaffolded but disabled in Phase A.** Migration `0002_rls_policies.sql` defines the policies; migration `0009_disable_rls_phase_a.sql` flips RLS off on every table. Reasoning: the app currently authenticates with the anon key and no Supabase Auth session (`persistSession: false`); with RLS enabled, `auth.uid()` returns NULL and every push/pull would be denied. The policies stay defined so Phase B activation is "apply 0002 + drop the hardcoded user_id" rather than "write the policies."

**Migration 0010 — namespace cloud tables under `buffr` schema.** Moved all 10 mirrored tables and the `get_server_time()` RPC from `public` into a dedicated `buffr` schema. Client at `src/services/sync/client.ts:47` sets `db: { schema: 'buffr' }`. Reasoning: two other apps will eventually share this Supabase project; schemas are the idiomatic Postgres way to keep them from colliding, and the move is metadata-only (no row rewrites, dependent objects follow the table). Activation required an out-of-band step (Supabase dashboard → Project Settings → API → "Exposed schemas" → add `buffr`) that surfaced as a silent sync stall because the orchestrator's `if (applied > 0 || fetched > 0)` log guard swallowed the PGRST106 errors. Rejected: `buffr_<table>` prefixes in `public` (would have required editing every `.from()` callsite and would have collided when other apps' table names overlap).

**Heuristic-first classifier, LLM as fallback.** `heuristicClassify` runs regex/keyword rules per todo line before deciding to call Haiku/4o-mini. Reasoning: latency, cost, offline tolerance. Confident matches skip the network round-trip entirely. Cost: regex rules drift and require periodic re-tuning; the heuristic and the LLM can disagree, which the system resolves by trusting the heuristic when its confidence is `high`.

**Provider-agnostic AI service layer (Claude default, OpenAI alternate).** 5 chains × 2 providers = 10 explicit branches; provider chosen via `services/ai/config.ts` read on every call. Reasoning: provider lock-in risk is real and the cost of abstracting two SDKs at the service-function level is small (each chain is one function with a branch). Rejected: a factory/strategy abstraction with a shared interface — the chains diverge enough (Claude's structured outputs vs OpenAI's `response_format: json_object`, plus the interpret chain wanting markdown from OpenAI without a JSON wrapper) that a lowest-common-denominator interface would have been more friction than the branches.

**Single-clip → multi-clip migration (clips_json) without breaking single-clip references.** Both `clip_uri` (legacy) and `clips_json` (array) coexist on `entries`. Reasoning: the multi-clip feature was added later; the old column round-trips through sync as the "primary clip" pointer for back-compat with rows authored pre-migration. Cost: every read path that wants "all clips" must fall back to `[{ uri: clip_uri, durationMs: clip_duration_ms }]` if `clips_json` is empty/null.

**User-overridden type is a permanent lock.** Once a user manually picks a thinking-mode for a todo, `user_overridden_type = 1` is set and AI re-classification cannot change the type. Reasoning: the classifier is sometimes wrong in ways the user notices; relinquishing user control to "the AI thinks better now" would undermine trust.

**Slug-as-local-canonical for threads.** A thread's `slug` is unique-per-user and is the matching key for `#tag` mentions in prose. Renames are local-only and would invalidate `thread_mentions` reconciliation against existing prose, so they go through the threads CRUD only — not as freeform edits. Rejected: slug-as-cloud-canonical (would require either rewriting prose on rename or storing an ID-to-slug map that the scanner consults).

**Composite-key local SQLite schema with `synced_at` column doubling as retry queue.** Failed pushes leave `synced_at` unstamped; the next push naturally retries the same rows because the dirty filter (`updated_at > synced_at`) still matches. No separate failure log, no in-flight-vs-queued state machine — the SQL predicate IS the retry queue. Reasoning: simplicity. The whole sync engine is ~12 files because every concept maps to one column.

**Manual-touch deviation in `services/threads/touch.ts`.** Writes a `thread_mentions` row with NULL `entry_id` AND NULL `todo_id` to mark a thread "done today" from the dashboard, even though the app-level invariant is "at least one of entry_id / todo_id is set." Reasoning: the dashboard tap needs to register without forcing the user to type `#tag` somewhere; the staleness math composes uniformly over all `thread_mentions` regardless of source. Documented as Principle 11 in `spec.md`.

**`expo-secure-store` for API keys + bootstrap flags + backfill gates.** Eight or so SecureStore keys total: `anthropic_api_key`, `openai_api_key`, `ai_provider`, `supabase_url`, `supabase_anon_key`, `cloud_initial_push_done`, and per-feature backfill flags. Reasoning: Android Keystore-backed; survives uninstall on some Android versions; right primitive for "small auth-adjacent values."

**Release builds signed with the debug keystore.** `android/app/build.gradle` declares `release { signingConfig signingConfigs.debug }`. Reasoning: this is a sideloaded personal app with no Play Store presence; debug-keystore signing means no manual `keytool` ceremony per build and no risk of misplacing a production keystore.

**No automated test suite.** Strict TypeScript (`npx tsc --noEmit` is required before commit) is the only automated gate. Reasoning: single-developer pace + the cost of writing tests for a UI-heavy RN app + a feature surface that's been changing weekly outweighs the regression-catch value at this stage. Manual end-to-end on the connected Android device after each meaningful change is the substitute.

## 7. Risks and rough edges

**Silent sync failures.** `src/services/sync/orchestrator.ts:72` only `console.log`s a pull result when `applied > 0 || fetched > 0`. Failed pulls (returned `{fetched: 0, applied: 0, error: '...'}`) are stored in `sync_meta.last_error` but emit nothing to logcat. The 2026-05-19 schema move surfaced this exactly — sync was broken for an hour with zero logcat signal until a `curl` test against Supabase REST revealed the missing "Exposed schemas" dashboard step. The `sync_meta` ledger holds the truth; the console doesn't.

**Aging clip files in the old Android sandbox.** The loopd→buffr Android applicationId change (`com.anonymous.loopd` → `com.anonymous.buffr`) created a new sandbox. ~183 clips that lived in the old sandbox are now invisible to buffr — every cold start logs "clipMigration source missing, skipping" warnings for each. DB rows reference paths that no longer resolve. Either the old loopd APK still exists on the device (with the old files) or those clips are gone unless they were saved to DCIM.

**Phase A user_id is hardcoded across the entire push/pull layer.** `PHASE_A_USER_ID = '00000000-0000-0000-0000-000000000001'` in `src/services/sync/client.ts:6`. Every mapper file under `src/services/sync/tables/*` stamps it on writes. The Phase B switch isn't one config flip — it's "find every stamping site and replace with `auth.uid()`," and the existing rows in cloud all carry the hardcoded UUID and would need re-stamping.

**Release signing uses the debug keystore.** Every release APK is signed with `android/app/debug.keystore` (which is the auto-generated, well-known Android debug key). The app cannot be uploaded to Play Store as-is; switching keystores later would mean every existing install (which is currently one device) would need to uninstall + reinstall.

**Android-only.** `app.json` declares `platforms: ['android']`; iOS isn't supported. Adding iOS would require an Apple developer account, a separate native bundle, ffmpeg-kit iOS variant, and re-validating every native module.

**`react-native-reanimated` 4.x + `react-native-worklets` 0.7.x are pre-1.0.** The animation stack is functional but on a moving target; major-version bumps in Expo SDK 56+ are likely to come with breaking changes the codebase will need to absorb.

**`@wokcito/ffmpeg-kit-react-native` is a community fork.** The upstream `arthenica/ffmpeg-kit` was archived. `@wokcito/...` continues to publish builds against newer React Native versions, but it's a single-maintainer fork. Replacement would mean either a different ffmpeg wrapper (none with comparable RN support today) or rolling a custom native module.

**No automated tests.** A regression in any of the scanners (`scanTodos`, `scanNutrition`, `scanThreads`), the reconciler (`reconcileMeta`), or the sync engine wouldn't be caught until the user notices broken data in the UI. The `tsc --noEmit` gate catches type-shape regressions but not behavioral ones.

**The `sync_deletions` table + `notion_page_id` columns are dead but present.** Read paths don't consult them; they round-trip through cloud sync because the mappers still mention them. Future readers will wonder what they're for; the answer is "ex-Notion outbox, deleted feature, not yet vacuumed."

**The `clips_json` / `clip_uri` coexistence.** Every read site that wants all clips has to fall back to `[{ uri: clip_uri, durationMs: clip_duration_ms }]` if `clips_json` is empty. A new contributor would miss the fallback and ship a bug visible only on old rows.

**`scripts/db-migrate.mjs` and dashboard SQL editor are diverging paths.** Migrations 0001–0009 went through the Node runner; 0010 went through the Supabase dashboard SQL editor because of the `DO $$ … $$` block. Both work, but there's now no single canonical "how migrations get applied" answer.

**Supabase free tier is the operational backstop.** Cloud sync currently costs $0. Phase B (multi-user) plus clip-file backup plus an active user base would push past the free tier quickly. Pricing decisions are deferred.

**The `interpret` AI chain is the only one whose output is not persisted.** Modal-only — if the user closes the modal, the markdown is gone, and re-running the chain costs another LLM call. Intentional (per the spec) but a rough edge if you want to refer to a past interpret session.
